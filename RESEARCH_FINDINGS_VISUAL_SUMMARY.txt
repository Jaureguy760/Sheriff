QUESTION 1: Why is rust-htslib only 1.9x faster than pysam?
================================================================================

Answer: Decompression (60% of BAM I/O time) uses shared C code both must use.

    Cost breakdown of BAM reading:
    ┌─────────────────────────────────────────────┐
    │ BGZF Decompression (HTSlib C):  60% │ Shared │
    │ BAM Parsing (HTSlib C):           20% │ Shared │
    │ Tag Extraction + Python overhead: 20% │ rust-htslib optimizes this
    └─────────────────────────────────────────────┘
    
    pysam:        100 units
    rust-htslib:  121 units → 1.9x speedup on the 40%
    Theoretical:  30 units → would need 3.3x (impossible)

Conclusion: Can't get much better without changing decompression itself.

QUESTION 3: Can we do parallel BAM reading? How?
================================================================================

Answer: Technically YES. Practically NOT WORTH IT.

    BGZF Format ALLOWS Parallelism:
    ┌──────────────────────────────────────────────────────────┐
    │ [Block 1][Block 2][Block 3][Block 4]...[Block N]        │
    │   ↓         ↓         ↓         ↓              ↓         │
    │ Each block can be decompressed independently!           │
    │                                                           │
    │ Virtual offset = (block_offset << 16) | within_block    │
    │                                                           │
    │ This enables parallel decompression by design!          │
    └──────────────────────────────────────────────────────────┘

    Tools Already Doing This:
    - Sambamba:   Uses multi-threaded decompression (2-6x faster)
    - quickBAM:   Memory-mapped + parallel (published paper)
    - ompBAM:     C++ OpenMP-based parallel reading

    BUT HTSlib Deliberately Doesn't:
    - Simplifies memory management
    - Works well for streaming (their main use)
    - Single-threaded decompression is a feature, not a bug

    If We Implemented Parallel BAM:
    Time required:     12-16 hours
    Expected benefit:  3-6% total pipeline
    Architecture risk: High
    Maintenance burden: Ongoing
    ROI:               0.2-0.5 pp/hour (MEDIOCRE)

Recommendation: Don't implement. Better alternatives exist with better ROI.

QUESTION 4: Zero-copy BAM data - is it possible?
================================================================================

Answer: NO. Decompression requires memory allocation, can't be avoided.

    Current Data Flow:
    BAM File (compressed)
        ↓ [Decompression - MUST copy to memory]
    Memory buffer (uncompressed)
        ↓ [Parsing]
    BAM Record objects
        ↓ [Tag extraction]
    Python objects
        ↓ [Dict/set storage]
    Final results

    PyO3 Buffer Protocol Limits:
    ✓ Can zero-copy byte slices from Rust to Python
    ✗ Cannot return lifetime-tied references (GIL conflicts)
    ✗ Cannot eliminate decompression copies (it's math)
    ✗ Cannot avoid Python object overhead

    Reality Check:
    - Tag extraction: ~1-2KB per read
    - Decompression: ~10-50x that size
    - Zero-copy only helps the tiny part!

    If We Did It Anyway:
    Time required:      8-12 hours
    Copy reduction:     10-15%
    Total speedup:      1-2%
    ROI:                0.08-0.25 pp/hour (TERRIBLE)

Recommendation: Skip completely. Addresses wrong bottleneck.
